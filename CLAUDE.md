# TD-MD Core — руководство проекта (для Claude Code)

GPU-движок молекулярной динамики на основе **метода декомпозиции времени (Time Decomposition, TD)** из кандидатской диссертации Андреева В.В. (2007). Цель: воспроизвести и развить подход диссертации — от подготовки структуры до запуска расчётов и анализа.

## Источник истины
1. `source/time_decomposition.docx` — диссертация (физика, метод, параметры). **При любом сомнении сверяйся с ней и ссылайся на разделы.** Для повседневной работы используй `source/time_decomposition.md` — полнотекстовую версию с **формулами, транскрибированными в LaTeX** (все 52 нумерованных уравнения; нерасшифрованные инлайн-символы помечены `⟨imageNNN⟩`). Старый `time_decomposition.txt` — сырой экстракт без формул, оставлен для grep-совместимости.
2. Сверка ключевых формул (ур. 33, 48, 62, K2, Морзе) с кодом выполнена 2026-06-11: `docs/_meta/FORMULA_VERIFICATION_2026-06-11.md`. Там же — маппинг обозначений (дисс. C1/C2/С1 ⇄ код `C_buf`/`C1`). Буквы зон приведены к диссертационным во всём проекте 2026-06-11.
3. `docs/` — проектные инструкции, выведенные из диссертации. Тег `[ENG]` помечает инженерные расширения (GPU/CUDA и т.п.), которых в диссертации нет и которые не были ею валидированы.

## КРИТИЧЕСКИЕ инварианты (нарушать нельзя)
- **Единицы:** только LAMMPS `metal` (эВ, Å, amu, ps). См. `docs/TD_MD_Core_Units_v1_0.md`.
- **Автомат зон:** цикл строго `o → d → w → c → p → o` (o=свободна, d=данные приняты, w=частичные силы, c=расчёт, p=готова к передаче) — **буквы строго по диссертации** (переименовано 2026-06-11; до этого проект использовал перестановку c/p/o — см. `FORMULA_VERIFICATION_2026-06-11.md` §6, историю не путать с текущей нотацией). Детали и инварианты — `docs/TD_MD_Core_ZoneFSM_v1_0.md`.
- **Причинность:** ни один атом не пересекает буфер $R_{buf}=v_{max}\Delta t C$ ($C\ge1$; ур. 33, в диссертации коэффициент назван C1, в конфиге — `C_buf`) за шаг; иначе HALT + rescue.
- **$\Delta t$/$v_{max}$ локальны** для узла — никаких глобальных `Allreduce` на каждом шаге (рушит конвейер).
- **Точность:** глобальные координаты FP64; накопление сил FP64 с фиксированным порядком (детерминизм); ядра — `template<typename Real>` (одна кодовая база, FP32/FP64).
- **Ансамбль v1:** только NVE. NVT/NPT — backlog.
- **Запрещено:** ReaxFF, PME, QEq (глобальная редукция несовместима с асинхронным TD).

## Эталонные данные
`reference_data/` — золотой датасет Al/Морзе (72 атома, PBC), силы проверены конечными разностями. Самодостаточен: базовые тесты и CI **не требуют LAMMPS**. Используется в `Test_0_Step` и тесте детерминизма.

## Технологический стек
C++20, CUDA NVCC через слой HAL (без Kokkos), MPI, CMake, Google Test, YAML-cpp. Профилирование NVTX; деплой Apptainer/Spack.

## Конвенции
- Пути в `config.yaml` — относительно корня репозитория (CWD движка), напр. `reference_data/al_fcc_72.data`.
- Кросс-ссылки между документами — по имени файла внутри `docs/`.
- `seed` фиксируется в конфиге; прогоны воспроизводимы.
- Каждый новый модуль — с юнит-тестом (см. `Skeleton_Interfaces §9`).
- VerifyLab/liblammps — опционально за CMake-флагом `-DWITH_LAMMPS=ON`; не делать жёсткой зависимостью CI.

## Текущий статус (2026-06-11)
Выполнены вехи **M0–M2.5** (критерии приёмки зелёные, см. `Roadmap`):
- **M0** — walking skeleton: CPU, O(N²) Морзе, velocity-Verlet NVE, `Test_0_Step` (силы ≤1e-6 эВ/Å vs golden) + `Test_NVE_Drift`.
- **M1** — `ZoneFSM` (чистая логика): таблица переходов, INV-3, seed S₁, чёт/нечёт I/O-порядок; `Test_Zone_FSM`.
- **M2** — буфер причинности (ур. 33) + авто-шаг C1/K2/C3 + HALT/rescue; `Test_Buffer`.
- **M2.5** — аналитический occupancy-зонд (Tier-1, без CUDA): отчёт `docs/_meta/occupancy_probe_2026-06-06.md`; вывод — одиночная тонкая зона недогружает GPU, нужен батчинг ~зон/стрим; `Test_Occupancy`.

**Следующая веха — M2.6** (тепловая инициализация: Maxwell по `seed`, чтение Velocities, тест импульса), затем M3 (clustering, Tier-split) → M3.5 (TimeConveyor-CPU) → M4 (порт на CUDA). План пересмотрен по второму арх-ревью: `docs/_meta/ARCH_ROADMAP_REVIEW_2026-06-11.md` (B1 фикс-точечное накопление, B2 Δt-handoff, B3 PBC-замыкание, B4 k шагов/узел). Переход к следующей вехе — только после зелёного критерия приёмки текущей (`Roadmap`). Сборка: `cmake -S . -B build -G Ninja && cmake --build build && ctest --test-dir build` (5 тестов).
