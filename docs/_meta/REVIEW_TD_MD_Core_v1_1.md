## 3. Найденные расхождения (по убыванию серьёзности)

### 🔴 КРИТИЧЕСКОЕ — конечный автомат зон в ТЗ перепутан
Диссертация (Глава 2.1, рис. 19, стр. 560–610) определяет 5 типов так:
`o` — рассчитана/готова к передаче · `p` — в процессе расчёта · `w` — частичное взаимодействие учтено · `d` — данные есть, не используются · `c` — свободна (нет данных).

**ТЗ §2.3 переназначило буквы наоборот:** «Свободна (o) → Ожидание данных (d) → Взаимодействие (w) → Расчёт (c) → Готова к отправке (p)». То есть `o`↔`c`, `c`↔`p` перепутаны местами. Это **прямо противоречит и диссертации, и файлу TDfull** в том же комплекте. Для ИИ-агента, который будет писать `TimeConveyor`, это гарантированная логическая ошибка в ядре. **→ Исправлено.**

Правильный цикл: **`c → d → w → p → o → c`** (5 переходов = 5 правил смены типа из §2.1).

### 🟠 СРЕДНЕЕ — неполный цикл зон в Rationale
Rationale указывал «`o → d → w`» — неверное начало (старт всегда `c`) и обрыв цикла. **→ Исправлено** на полный `c → d → w → p → o → c`.

### 🟠 СРЕДНЕЕ — тезис о расхождении траекторий противоречит результату диссертации
Rationale §8 утверждает, что траектории «полностью расходятся за 1000 шагов». Но диссертация (Глава 3.6, рис. 43–44) показала **строго нулевое** отклонение 1-проц vs 4-проц на 25 900-м шаге — метод TD не меняет порядок суммирования. Расхождение по Ляпунову относится к сравнению с *другим* движком (LAMMPS) или к недетерминированному FP32. **→ Добавлено уточнение** + рекомендация сохранить детерминированный режим FP64 как регрессионный тест.

### 🟡 НИЗКОЕ — заглушки цитат `[cite: 1]`
В TZ и Rationale почти все ссылки — заглушка `[cite: 1]`, не указывающая на раздел. Нарушает требование проекта «явно ссылаться на разделы». **→ Добавлено Приложение А** (трассируемость к реальным главам/формулам) в TZ.

### 🟡 НИЗКОЕ — отсутствовали параметры эталона
Не был зафиксирован эталонный потенциал для Test_0_Step. **→ Добавлены** в TDfull: масса Al, $D=0.29614$ эВ, $\alpha=1.11892$ Å⁻¹, $r_0=3.29692$ Å, 72 атома, ГЦК (Глава 3.5).

### 🟡 НИЗКОЕ — «2 полушага» Варлета
ТЗ §3.1 говорит о «2 полушагах» (kick-drift-kick). Диссертация даёт совмещённую скоростную форму (ур. 15–16). Формы эквивалентны, но реализация «2 полушага» требует хранить $f(t+\Delta t)$ между полушагами — стоит отметить явно (несрочно).

## 5. Рекомендации для разработки в Claude Code (что усилить далее)

1. **Сделать автомат зон центральным артефактом.** Самая новая и сложная часть метода — переходы `d→w` (момент «попадания в сферу $R_{cut}$») и учёт частичных сил в состоянии `w`. Стоит вынести в отдельный документ формальную таблицу «состояние × событие → состояние + инвариант» и привязать к ней Спринт 3.

2. **Развести два режима валидации.** (а) *Детерминированный* FP64: TD-версия = однопоточная побитово (повторяет результат диссертации, Гл. 3.6). (б) *Статистический* (RDF ≥ 99.5 %) против LAMMPS. Сейчас в файлах есть только (б); (а) — более строгий и дешёвый регрессионный тест.

3. **Явно пометить нефизические допуски как инженерные.** Пороги `Test_0_Step ≤ 1e-6`, `NVE ≤ 1e-5` — это выбор разработки, а не из диссертации; так и подписать, чтобы агент их не считал «истиной из первоисточника».

4. **Зафиксировать границы scope по потенциалам.** Диссертация валидировала только парный Морзе. EAM/FS/MEAM/ML — обоснованное расширение метода, но без экспериментального подтверждения в первоисточнике. Базовую корректность ядра проверять на Морзе/Al; остальные потенциалы — как отдельные, независимо тестируемые модули.

5. **Проверить FP32 local offsets.** Хранение смещений в FP32 относительно границы зоны корректно, пока ширина зоны ограничена (точность ~1e-5 Å при offset ~100 Å). Стоит задать верхний предел ширины зоны как инвариант.

6. **Связать спринты с трассируемостью.** Каждый спринт ТЗ §9 полезно снабдить ссылкой на раздел диссертации и критерий приёмки из §7 (VerifyLab).
