# ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ПРОМЫШЛЕННАЯ ВЕРСИЯ v1.1)
## Разработка коммерческого GPU-вычислительного движка молекулярной динамики на основе метода декомпозиции времени (TD-MD Core)

---

## 1. Общие сведения и назначение продукта (Product Vision)

* **Наименование системы:** Промышленный вычислительный комплекс молекулярной динамики с временной конвейеризацией (TD-MD Core).
* **Назначение:** Высокопроизводительное моделирование кристаллических структур, многокомпонентных сплавов, процессов радиационного повреждения и межфазных границ на гетерогенных вычислительных системах (GPU-кластерах).
* **Ключевое отличие от аналогов:** Использование инвертированной схемы параллелизма — **декомпозиции времени (Time Decomposition, TD)**[cite: 1]. Распределение последовательных временных шагов по кольцевому конвейеру GPU для локального расчета сложных многочастичных сил без сетевых задержек[cite: 1].
* **Целевое окружение разработки:** Claude Code (CLI-среда автоматизированной разработки). Локальная разработка ведется на RTX 5080 с последующим деплоем на HPC-кластер.
* **Стек технологий:** C++20, CUDA NVCC (через слой HAL), MPI, CMake, Google Test, YAML-cpp.

---

## 2. Глобальная архитектура (Core Architecture)

Движок функционирует как автономное приложение (Standalone Engine), построенное на принципах изоляции физики и минимизации синхронизаций с CPU.

### 2.1. Концепция "GPU-Resident"
* Весь главный цикл симуляции (интегратор, пересчет буфера, поиск соседей, расчет сил) выполняется СТРОГО на GPU. 
* Запрещены функции явной синхронизации хоста (`cudaDeviceSynchronize`) внутри цикла. Синхронизация между шагами конвейера реализуется аппаратно через `cudaEvent_t`. CPU работает в неблокирующем режиме как диспетчер.

### 2.2. Слой аппаратной абстракции (C++ HAL)
* Физика потенциалов отделяется от API конкретного вендора через легковесный заголовочный слой `hal.hpp`. Абстрагируются синхронизация потоков (`HAL_SYNC_THREADS`), локальная память (`HAL_SHARED_MEMORY`) и атомарные операции. 
* Запрещено использование тяжелых сторонних фреймворков (типа Kokkos).

### 2.3. Временной кольцевой конвейер (Multi-Stream / Multi-Node)
* Моделируемый объем разделен на расчетные зоны ($S_1 \dots S_n$) вдоль координатной оси[cite: 1].
* Вычислительные узлы организованы в логическое кольцо[cite: 1]. 
* **Локальная эмуляция (RTX 5080):** Узлы эмулируются массивом независимых очередей `cudaStream_t` внутри одного GPU.
* **Кластерный режим:** Узлы мапятся на физические GPU. Передача данных осуществляется через GPUDirect P2P (NVLink) или MPI, полностью маскируясь вычислениями.
* **Цели режимов (не смешивать).** Одно-GPU эмуляция (`streams`) валидирует **только логику** автомата зон, причинность и детерминизм — между стримами нет сети, поэтому latency hiding и метрика **P2P Masking %** там недостоверны. Тезис о производительности (линейный рост, >95%, перекрытие передачи) проверяется **только** в multi-GPU режиме. Метрики производительности к одно-GPU режиму не предъявляются.
* **Конечный автомат расчётных зон** (диссертация, Глава 2.1, рис. 19, стр. 560–610; буквы приведены к диссертационным 2026-06-11, см. `_meta/FORMULA_VERIFICATION_2026-06-11.md` §6). Каждая зона имеет динамически изменяемый тип:

| Тип | Состояние зоны |
|-----|----------------|
| `o` | свободна: данных нет, память очищена (готова к приёму) |
| `d` | данные приняты, но ещё не используются в расчёте |
| `w` | пограничная: в матрице сил учтено частичное взаимодействие с соседней рассчитываемой зоной |
| `c` | рассчитывается в данный момент |
| `p` | рассчитана, данные больше не нужны на этом шаге, готова к передаче следующему узлу |

  Полный жизненный цикл — 5 переходов (диссертация §2.1, пять правил смены типа):
  `o` →(приём данных от предыдущего узла)→ `d` →(попадание в сферу потенциала $R_{cut}$ активной зоны)→ `w` →(начало расчёта зоны)→ `c` →(окончание расчёта)→ `p` →(передача данных следующему узлу)→ `o`.
  > **Важно для реализации.** Тип `w` означает, что силы атомов зоны накоплены лишь частично (учтён вклад одной соседней зоны по 3-му закону Ньютона). Движок обязан вести флаг «частичности» матрицы сил, иначе причинность нарушится. Зоны `p` и `c` предпочтительно назначать одного размера для синхронного приёма/передачи (диссертация §2.3: «зоны $s_p$ и $s_c$ … одинакового размера»); минимальный размер зоны типа `w` определяется $R_{cut}$ (диссертация §2.3, стр. 763).

## 3. Физико-математическое ядро (Micro-Architecture)

### 3.1. Интегратор и закон причинности
* Интегрирование выполняется по скоростной форме алгоритма Варлета (Гл. 1.2, ур. 15–16). Реализация «2 полушага» (kick-drift-kick) эквивалентна совмещённой форме, но требует хранить $f(t{+}\Delta t)$ между полушагами.
* **Ансамбль (scope v1):** реализуется **только NVE** (как в диссертации, Гл. 3). Термостаты/баростаты (NVT/NPT, Нозе-Гувер) — в backlog; $K_2$ ограничивает прирост температуры за шаг, но термостатом не является.
* **Динамический шаг $\Delta t$ и $v_{max}$:** Максимальная скорость атома ищется через `Parallel Reduction` на GPU. **$v_{max}$ и $\Delta t$ локальны для узла** (узел прогоняет весь модельный шаг, Гл. 3.5) — глобальный `Allreduce` на каждом шаге не вводить, иначе рушится асинхронный конвейер; для следующего шага берётся консервативный $v_{max}$ предыдущего прохода.
* **Буферная зона:** Пересчитывается динамически ($R_{buf} = v_{max} \cdot \Delta t \cdot C_{buf}$, $C_{buf}\ge1$) для гарантии нерушимости причинности на границах зон (Гл. 2.1, ур. 33).

### 3.2. Паттерн "Warp-Aligned Clustering"
* Вместо классических списков соседей атомы внутри зоны сортируются пространственным хэшем и упаковываются в фиксированные блоки по 32 атома (размер CUDA Warp). Это гарантирует коалесцентный доступ к памяти и исключает расхождение потоков (Thread Divergence).
* **Роль сетки ячеек (уточнение).** «≤1 атом на ячейку» (диссертация Гл. 3.1, ≈2.33 Å для Al) — это схема пространственной сортировки для MIMD-машин 2007 г., а **не** структура хранения на GPU. На GPU сетка используется только для пространственной сортировки (Z-order/Morton), результат которой упаковывается в 32-атомные варп-кластеры. Буквальные почти-пустые ячейки в памяти GPU не материализуются (это дало бы плохую коалесценцию).

## 4. Управление памятью и точностью (Mixed Precision)

1. **Memory Layout:** Структура данных реализуется строго по паттерну **SoA** (Structure of Arrays).
2. **Global Coordinates:** Глобальные координаты атомов хранятся в формате `double` (FP64).
3. **Local Offsets:** Внутри зоны и при передаче по сети координаты конвертируются в локальные смещения (`float`, FP32) относительно левой границы зоны (экономия 50% трафика). **Только в `production_mixed`:** в режиме `deterministic_fp64` offsets и транспорт — FP64 (конвертация теряет биты и ломает побитовый детерминизм; B5).
4. **Force Evaluation:** Расчет уравнений сил выполняется в `float` (FP32).
5. **Force Accumulation (Детерминированность):** Накопление сил на атом — порядко-независимое. **Основной путь — фиксированная точка**: конвертация вкладов в `long long int` (масштаб ~2⁴⁰ на эВ/Å) и целочисленное атомарное сложение — оно ассоциативно, поэтому результат побитово воспроизводим при любом порядке потоков и в обоих режимах точности (B1; как в LAMMPS GPU/Anton). Аппаратный `atomicAdd(double)` имеет нефиксированный порядок выполнения и используется только как референс для сверки.
6. **Единый код для двух режимов точности (правило).** Чтобы не плодить два набора ядер, расчётные ядра шаблонизируются по типу `template<typename Real>` (одна кодовая база, инстанциации `float` и `double`). Режим `deterministic_fp64` (валидация, повторяет нулевое отклонение Гл. 3.6) и боевой `production_mixed` — это две инстанциации одного кода, а не разные ядра. Управляется через `config.yaml` (`precision.mode`) и CMake-опцию.

## 5. Модуль потенциалов взаимодействия (Physics Suite)

Движок поддерживает сплайновую интерполяцию табличных данных.

### 5.1. Поддерживаемые потенциалы
* **Morse:** Базовый аналитический парный потенциал для отладки ядра[cite: 1].
* **EAM (Embedded Atom Method):** Для плотных металлов. Таблицы (формат `.eam.alloy`) загружаются через `Texture Objects` или `Constant Memory`.
* **FS (Finnis-Sinclair):** Выделяется в **отдельный модуль и CUDA-ядро** из-за специфичной математической формы кросс-взаимодействий при расчете многокомпонентных сплавов.
* **MEAM:** Учет направленных связей. Агрессивное кэширование координат соседей в `Shared Memory`.
* **ML-Potentials Layer:** Абстрактный интерфейс для нейросетевых потенциалов с использованием Tensor Cores (TF32/BF16).

### 5.2. Ограничения (Out of Scope)
* **СТРОГО ЗАПРЕЩЕНО** внедрение реактивных потенциалов (ReaxFF) и электростатики с глобальным выравниванием зарядов (QEq, PME). Глобальная редукция матриц на каждом шаге нарушает локальную причинность метода TD и разрушает асинхронный конвейер.

## 6. Интерфейс пользователя (UX/UI) и Телеметрия

### 6.1. Ввод и конфигурация
* Конфигурация движка задается через современный `config.yaml`. Геометрия парсится из `read_data` (формат LAMMPS).
* **Предстартовый чек-лист:** Вывод в консоль доступных GPU, топологии конвейера и предупреждений о режиме эмуляции.

### 6.2. Живой дашборд (Live CLI Dashboard)
* **Прогресс:** Процент выполнения, точный прогноз ETA.
* **Live Stats:** Инварианты ($E_{pot}$, $E_{kin}$, $E_{full}$, Температура).
* **TD Health (Метрики конвейера):** Текущий $R_{buf}$, % загрузки GPU, **P2P Masking %** (показатель перекрытия сетевых задержек).

### 6.3. I/O и Защита от сбоев (Graceful Exit)
* **Асинхронный I/O:** Запись траекторий (`.lammpstrj`) и телеметрии выполняется в фоновом CPU-потоке.
* **Rescue Dump:** При перехвате CUDA-ошибки (OOM) или нарушении причинности ($v_{max} > R_{buf}$) движок останавливает конвейер и сбрасывает спасательный дамп `rescue.xyz`.

## 7. Подсистема автоматической валидации (VerifyLab)

Изолированная подсистема для контроля хаотической неустойчивости и кросс-валидации.
* **LammpsRuntimeBridge:** Интеграция с `liblammps.so` (C-API) для прямого обмена массивами.
* **Test_0_Step:** Сравнение вычисленных сил в нашем движке и LAMMPS на нулевом шаге ($\max|F_{lammps} - F_{td}| \le 10^{-6}$).
* **Test_NVE_Invariant:** 50 000 шагов NVE. Проверка отсутствия дрейфа $E_{full}$ (флуктуации $\le 10^{-5}$) и сохранения импульса.
* **Test_Statistical:** Сравнение функции RDF длинных траекторий с результатами LAMMPS (совпадение $\ge 99.5\%$).

## 8. Инженерные практики и HPC-развертывание (DevOps & MLOps)

* **8.1. Профилирование NVTX:** Код ядра должен быть размечен маркерами NVIDIA Tools Extension (`nvtxRangePush`). Это необходимо для визуализации перекрытия передачи данных (P2P Transfer) и вычислений в Nsight Systems.
* **8.2. Контейнеризация:** Проект должен включать `Apptainer.def` (или `Singularity.def`) для мгновенного развертывания на HPC-кластерах, а также конфигурацию для пакетного менеджера **Spack**.
* **8.3. Управление Seed:** Все ГПСЧ инициализируются глобальным `seed` из конфигурации для обеспечения повторяемости.
* **8.4. Пайплайн CI/CD:** Наличие `.github/workflows/ci.yml` (или аналога) для автоматического линтинга, сборки под GCC/Clang и запуска статических тестов `VerifyLab` при каждом коммите.

## 9. Пошаговые инструкции для ИИ-агента (Спринты)

Агент выполняет разработку строго по этапам. Переход на следующий этап — только после успешного тестирования текущего.

* **Спринт 1 (Фундамент & Инфраструктура):** Настроить CMake, CI/CD. Создать Apptainer.def. Реализовать `hal.hpp`. Создать структуры SoA (Mixed Precision с детерминированным накоплением). Реализовать YAML-парсер.
* **Спринт 2 (Вычислительное ядро):** Написать интегратор Варлета[cite: 1], $v_{max}$ (Parallel Reduction). Реализовать Warp-Aligned Clustering. Добавить NVTX-маркеры в ключевые функции.
* **Спринт 3 (Конвейер Времени):** Реализовать `TimeConveyor`. Настроить конечный автомат зон[cite: 1], аппаратную синхронизацию через `cudaEvent_t`. **Реализовать поддержку Multi-GPU. Для локальной отладки использовать режим P2P-эмуляции через `cudaStream_t` на одном GPU.**
* **Спринт 4 (Физика & VerifyLab):** Развернуть `VerifyLab` с мостом к `liblammps`. Написать ядро потенциала Морзе[cite: 1], пройти Test_0_Step. Написать независимые ядра для EAM и FS.
* **Спринт 5 (UX/UI & HPC Деплой):** Интегрировать ANSI CLI Dashboard. Реализовать асинхронную запись логов и Rescue Dump. Провести финальную отладку на GPU-кластере через реальную шину NVLink/MPI.
