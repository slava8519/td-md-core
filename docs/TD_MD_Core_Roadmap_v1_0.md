**Зачем.** Спринты ТЗ §9 слишком крупные для пошаговой ИИ-разработки (например «Спринт 3: конвейер + автомат + события + multi-GPU» — это месяцы без промежуточных ворот). Здесь — мелкие вехи `M0…M7`, каждая с **критерием приёмки**, привязанным к VerifyLab. Правило: переход к следующей вехе только после зелёного критерия текущей.

**Ключевой принцип — Walking Skeleton (A5).** Сначала сквозной минимальный прогон без TD-сложности, затем наращивание. Это рано вскрывает ошибки единиц/сил/интегратора, пока их дёшево чинить.

## ✅ M0 — Walking Skeleton (сквозной минимум) — DONE 2026-06-06
- **Цель:** запускаемый бинарь без TD-конвейера: 1 зона, 1 стрим, Морзе, NVE, FP64, фиксированный $\Delta t$.
- **Делается:** CMake (C++20), парсер `config.yaml` (мин. набор), чтение `al_fcc_72.data`, наивный $O(N^2)$ расчёт Морзе, скоростной Варлет, запись `.lammpstrj`.
- **Критерий приёмки:** `Test_0_Step` — силы совпали с `reference_data/reference_forces.csv` ($\le10^{-6}$ эВ/Å); 100 шагов NVE без NaN; энергия дрейфует слабо.
- **Закрывает риски:** единицы (A1), базовая корректность сил/интегратора.
- **Соответствие ТЗ:** часть Спринта 1 + 2.

## ✅ M1 — Конечный автомат зон (изолированно) — DONE 2026-06-06
- **Цель:** `ZoneFSM` как чистая логика, без GPU.
- **Делается:** enum состояний/событий, таблица переходов и guard'ы (`ZoneFSM §4`), флаг частичности (INV-3).
- **Критерий:** юнит-тесты всех допустимых переходов и отклонения недопустимых; цикл `o→d→w→c→p→o` воспроизводится; стартовый seed-случай $S_1$ и чётный/нечётный порядок (`ZoneFSM §7`).
- **Соответствие ТЗ:** ядро Спринта 3 (вынесено вперёд и изолировано).

## ✅ M2 — Буфер причинности и авто-шаг — DONE 2026-06-06 (семантика C3 уточнена 2026-06-11 по ур.62)
- **Цель:** защита причинности и динамический $\Delta t$.
- **Делается:** `R_buf=v_{max}\Delta t C` (ур. 33), `causality_ok` (INV-4), авто-шаг по $C_1/K_2/C_3$ (Гл. 3.3, 3.5), $v_{max}$ через Parallel Reduction; локальный $v_{max}$ без глобального барьера (A8).
- **Критерий:** тест буфера; искусственное $v_{max}\Delta t>R_{buf}$ → HALT + `rescue.xyz`; авто-шаг стабилен при $C_3=0.5$.
- **Соответствие ТЗ:** Спринт 2 (динамический шаг/буфер).

## ✅ M2.5 — Ранний occupancy-зонд (early go/no-go)  [ENG] — DONE 2026-06-06
- **Зачем (риск A2).** TD хочет тонкие зоны (~$R_{cut}$), GPU — десятки тысяч нитей. Одна тонкая зона может недогрузить чип так, что TD на одном GPU окажется *медленнее* наивной MD. Это make-or-break, и проверять надо **до** дизайна кластеризации M3, а не после: зонд даёт ранний go/no-go и числа (атомов/зону, нитей/SM, во сколько зон батчить), на которых строится M3.
- **Контекст железа.** Локально — Mac mini M4 Pro, NVIDIA/CUDA нет (допущение ТЗ про RTX 5080 неактуально). Поэтому зонд **двухуровневый**:
  - **Tier 1 — аналитический, локально без GPU.** Чистый C++/Python: по (`zone_width`, плотность, конфиг запуска ядра, оценка регистров/shared mem на нить) считает теоретическую occupancy и атомов/варпов на зону для эталонного Al и набора размеров; моделирует одиночную зону и батч из $k$ зон на стрим. Спецификации целевого GPU — из профиля устройства (или `cudaGetDeviceProperties` на железе), **догадки не хардкодить**.
  - **Tier 2 — эмпирический, на NVIDIA GPU/кластере (отложен).** `cudaOccupancyMaxActiveBlocksPerMultiprocessor` (теор.) + Nsight Compute `sm__warps_active…pct_of_peak` (achieved) на реальном ядре сил. За CUDA-гейтом; **CI не блокирует**.
- **Делается:** утилита `occupancy_probe` (Tier 1) + committed-отчёт `docs/_meta/occupancy_probe_<date>.md`: кривая occupancy vs ширина зоны / атомов на зону + рекомендация (atoms-per-block, число одновременных зон на стрим для насыщения цели).
- **Критерий приёмки:** Tier 1 собирается и прогоняется на M4 Pro; для эталонного Al и целевого GPU-профиля зафиксированы атомов/зону при `zone_width=R_cut`, предсказанная occupancy одиночной зоны и батча. Если одиночная зона ниже целевого порога загрузки — зонд **обязан** показать, что батчинг её восстанавливает; иначе явный red-flag и пересмотр стратегии зон до входа в M3. Tier 2 — отложенный пункт чек-листа (при доступе к GPU), не гейт CI.
- **Соответствие ТЗ:** Спринт 2 (нефункциональный критерий occupancy, A2). **Питает дизайн M3.**

## ✅ M2.6 — Тепловая инициализация (Maxwell, seed)  [B8] — DONE 2026-06-11
- **Цель:** реалистичные прогоны при $T>0$; задействовать `run.seed` (сейчас все тесты стартуют с $T\approx0$, где K2 и буфер почти не нагружены; диссертация тестирует тепловые режимы §3.5–3.6).
- **Делается:** максвелловское распределение скоростей при `run.init_temperature` (ГПСЧ от `seed`), обнуление импульса ЦМ; чтение секции `Velocities` в `reader_lammps`; `Test_NVE_Drift` при T=300 K + **проверка сохранения импульса**.
- **Критерий приёмки:** $T(0)$ = заданной (±5%); $|p_{ЦМ}|\le10^{-12}$; NVE-дрейф при 300 K зафиксирован численно; два прогона с одним `seed` побитово совпадают.
- **Закрывает:** B8; предусловие честной валидации M3.5/M4.
- **Итог (2026-06-11).** `tdmd/core/thermal.hpp`: собственный ГПСЧ SplitMix64 + Бокс—Мюллер (std::normal_distribution непереносим между stdlib — нарушал бы INV-9), фиксированный порядок выборки; обнуление импульса ЦМ (2 прохода — второй давит остаток округления с ~1e-12 до ~1e-13 на «неудачных» seed) и точный рескейл к T на $n_{dof}=3N-3$ (семантика LAMMPS `velocity create`). `reader_lammps` читает `Velocities` (сопоставление по id); `run.init_temperature` в конфиге (>0 — Maxwell от seed, перекрывает Velocities; 0 — скорости из файла). Измерено (golden Al-72): T(0)=300 K точно; $|p|\le2\cdot10^{-13}$ amu·Å/ps по всем компонентам (5 seed); NVE-дрейф 300 K, dt=1 фс, 100 шагов: rel=7.1e-4 (тест-порог 3e-3); импульс после 100 шагов ≤6e-13; два прогона движка с одним seed — `cmp` побитово идентичны. `config_auto.yaml` переведён на тепловой старт 300 K: авто-шаг под нагрузкой опускает dt 5 фс → 0.53 фс (K2/C1 реально работают). Тесты: `Test_Thermal` (7 кейсов) + `Test_NVE_Drift.EnergyAndMomentumConservedAt300K`.

## M3 — Warp-Aligned Clustering + пространственная сортировка
- **Цель:** убрать $O(N^2)$, обеспечить коалесценцию.
- **Делается:** сетка-сортировка (Z-order) → упаковка в 32-атомные кластеры (роль сетки — A9); **список пар кластеров со skin-слоем** (`neighbor.skin`; аналог таблиц Варлета: слой $R_k-R_{max}$ валиден k шагов, см. ур. 32-контекст) с перестройкой по накопленному max-смещению (авто-шаг C1 ограничивает смещение ≤ C1·cell/шаг ⇒ кадрность перестройки предсказуема); NVTX-маркеры. Попутная гигиена (B10): строгая валидация конфига (enum'ы, диапазоны, неизвестные ключи — ConfigSchema §2), метки `xu yu zu` в писателе траекторий, юнит-тесты `config`/`reader_lammps`, HALT при перекрытии атомов.
- **Критерий (Tier-1, локально/CI):** силы кластерного пути ≡ силам прямого $O(N^2)$-пути (≤1e-12, внутренний кросс-тест) и golden (≤1e-6); таймингом подтверждён уход от $O(N^2)$ (N=10⁴ vs 10⁵); частота перестройки списка соответствует C1-оценке.
- **Критерий (Tier-2, на NVIDIA GPU, не гейт CI):** профиль Nsight без явного Thread Divergence; **achieved occupancy соответствует прогнозу M2.5** — не ниже целевого порога, либо зафиксировано расхождение модель/факт с объяснением (A2).
- **Соответствие ТЗ:** Спринт 2.

## M3.5 — TimeConveyor-CPU (логика TD без CUDA)  [ENG] [B6]
- **Цель:** снять конвейерные риски (B1–B3) на локальной машине до CUDA; CPU-кольцо даже ближе к MIMD-первоисточнику, чем стримы.
- **Делается:** z программных узлов (`std::jthread` + SPSC-очереди как заглушка `ITransport`), ≥4 зон, оркестровка через `ZoneFSM`, чёт/нечёт I/O; **Δt-handoff** (B2/ZoneFSM §6: Δt следующего шага решает узел-отправитель по v_max последнего завершённого прохода и передаёт с первой зоной; правило тождественно при z=1); **PBC-замыкание** (B3/ZoneFSM §7.2: циклические индексы зон, партнёр-`d` со сдвигом шага, удержание первых зон); **фикс-точечное накопление сил** (B1).
- **Критерий приёмки:** `Test_determinism` — побитовое совпадение координат **и скоростей** (методика Гл. 3.6) 1 узел vs z узлов, в т.ч. при `timestep.mode=auto`; `Test_NVE_Invariant` 50 000 шагов (дрейф $E_{full}\le10^{-5}$, импульс); PBC: NVE-дрейф на 4 зонах не хуже 1-зонного эталона; anti-deadlock ≥ 2z шагов; работает в CI.
- **Закрывает:** B1, B2, B3, B6; логическую часть A3. **Питает M4.**

## M4 — TimeConveyor: одно-GPU эмуляция (порт M3.5 на CUDA)
- **Цель:** перенос проверенной логики конвейера на массив `cudaStream_t`.
- **Делается:** `StreamTransport` (`cudaMemcpyAsync`), аппаратная синхронизация `cudaEvent_t` (INV-2); **фикс-точечное накопление на GPU** (B1; FP64-`atomicAdd` — только как референс сверки, его порядок не фиксирован); режим `deterministic_fp64` ⇒ FP64-offsets и транспорт (B5); Tier-2 пункты M2.5/M3 (deviceQuery, Nsight).
- **Критерий (главный):** те же тесты, что в M3.5 (побитовый детерминизм 1 стрим vs N стримов, NVE 50k, PBC, anti-deadlock) — на GPU; achieved occupancy зафиксирована и сопоставлена с прогнозом.
- **Закрывает:** A3 (single-GPU = валидация логики, не производительности).
- **Соответствие ТЗ:** Спринт 3.

## M5 — Multi-GPU (реальное кольцо)
- **Цель:** перенос на физические GPU без потери корректности; занятость чипа.
- **Делается:** `P2PTransport` (NVLink/`cudaMemcpyPeerAsync`) и `MpiTransport`; промежуточный режим «2 MPI-процесса на узле» как ступень достоверности (A4); **k временных шагов на узел** (`decomposition.ring.steps_per_node`, Гл. 3.4, ур. 49–51) — главный рычаг occupancy на multi-GPU (B4: внутри шага зоны причинно последовательны, поэтому в полёте у узла до k зон из его k шагов; ориентир k ≈ zones_to_saturate из M2.5-зонда), трафик ↓ в k раз.
- **Критерий:** корректность (детерминизм/NVE) сохраняется на ≥2 GPU; метрика **P2P Masking %** ненулевая; насыщение GPU при k>1 зафиксировано профилем; снят график strong/weak scaling (ориентир диссертации — линейный рост, >95%, Гл. 2.6).
- **Соответствие ТЗ:** Спринт 3 + 5.

## M6 — Physics Suite (многочастичные потенциалы)
- **Цель:** за пределы Морзе.
- **Делается:** EAM (отдельное ядро), затем FS; **дизайн партиального накопления электронной плотности ρ** поверх w-механизма зон (двухпроходность ρ→F на границах зон: w-вклад соседа — это и плотность, и силы; B-ревью, M6) ; production mixed-precision путь (шаблон `Real`, A6); интерфейс ML-потенциалов (заглушка).
- **Критерий:** EAM-силы и RDF длинной траектории vs LAMMPS — совпадение RDF $\ge99.5\%$ (`Test_Statistical`). Замечание: EAM/FS — `[ENG]`, в диссертации не валидировались (Гл. 3.2 — лишь обоснование возможности).
- **Соответствие ТЗ:** Спринт 4.

## M7 — UX / I/O / DevOps / HPC-деплой
- **Цель:** продуктовая зрелость.
- **Делается:** Live CLI Dashboard (ANSI, P2P Masking %, инварианты), асинхронный I/O в фоновом потоке, Rescue Dump на CUDA-ошибки, `Apptainer.def` + `spack.yaml`, полный CI (линт, сборка GCC/Clang, юнит-тесты VerifyLab).
- **Критерий:** CI зелёный на чистой машине (без LAMMPS — golden-тесты); деплой контейнера на кластере проходит smoke-тест.
- **Соответствие ТЗ:** Спринт 1 (CI/контейнеры) + 5.

## Сквозные требования (на каждой вехе)
- Каждый новый модуль — с юнит-тестом (см. `Skeleton_Interfaces §9`).
- `[ENG]`-решения не противоречат «источнику истины»; спорное — сверять с диссертацией.
- `seed` фиксирован; прогоны воспроизводимы.
- VerifyLab опционален за `-DWITH_LAMMPS=ON`; базовый CI — на golden-данных (A11).
- Локальная разработка — **без NVIDIA GPU** (Mac mini M4 Pro). CUDA-зависимые шаги (Tier 2 occupancy-зонда, потоки/события M4, multi-GPU M5) проектируются локально, исполняются и замеряются на кластере; CI и Tier-1-проверки GPU не требуют.
