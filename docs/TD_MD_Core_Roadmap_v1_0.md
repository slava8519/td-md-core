Здесь — мелкие вехи `M0…M7`, каждая с **критерием приёмки**, привязанным к VerifyLab. Правило: переход к следующей вехе только после зелёного критерия текущей.

**Ключевой принцип — Walking Skeleton (A5).** Сначала сквозной минимальный прогон без TD-сложности, затем наращивание. Это рано вскрывает ошибки единиц/сил/интегратора, пока их дёшево чинить.

## ✅ M2.5 — Ранний occupancy-зонд (early go/no-go)  [ENG] — DONE 2026-06-06
- **Зачем (риск A2).** TD хочет тонкие зоны (~$R_{cut}$), GPU — десятки тысяч нитей. Одна тонкая зона может недогрузить чип так, что TD на одном GPU окажется *медленнее* наивной MD. Это make-or-break, и проверять надо **до** дизайна кластеризации M3, а не после: зонд даёт ранний go/no-go и числа (атомов/зону, нитей/SM, во сколько зон батчить), на которых строится M3.
- **Контекст железа.** *(На момент M2.5)* локально — Mac mini M4 Pro, NVIDIA/CUDA нет. **Устарело 2026-06-11:** разработка переехала на Linux + RTX 5080 (sm_120); Tier-2 разморожен и выполнен в M2.7, измеренный профиль — `_meta/ARCH_AUDIT_GPU_2026-06-11.md` §0 (модель Tier-1 подтверждена с одной поправкой: 24 блока/SM, не 32). Исторический контекст зонда — зонд был спроектирован **двухуровневым**:
  - **Tier 1 — аналитический, локально без GPU.** Чистый C++/Python: по (`zone_width`, плотность, конфиг запуска ядра, оценка регистров/shared mem на нить) считает теоретическую occupancy и атомов/варпов на зону для эталонного Al и набора размеров; моделирует одиночную зону и батч из $k$ зон на стрим. Спецификации целевого GPU — из профиля устройства (или `cudaGetDeviceProperties` на железе), **догадки не хардкодить**.
  - **Tier 2 — эмпирический, на NVIDIA GPU/кластере (отложен).** `cudaOccupancyMaxActiveBlocksPerMultiprocessor` (теор.) + Nsight Compute `sm__warps_active…pct_of_peak` (achieved) на реальном ядре сил. За CUDA-гейтом; **CI не блокирует**.
- **Делается:** утилита `occupancy_probe` (Tier 1) + committed-отчёт `docs/_meta/occupancy_probe_<date>.md`: кривая occupancy vs ширина зоны / атомов на зону + рекомендация (atoms-per-block, число одновременных зон на стрим для насыщения цели).
- **Критерий приёмки:** Tier 1 собирается и прогоняется на M4 Pro; для эталонного Al и целевого GPU-профиля зафиксированы атомов/зону при `zone_width=R_cut`, предсказанная occupancy одиночной зоны и батча. Если одиночная зона ниже целевого порога загрузки — зонд **обязан** показать, что батчинг её восстанавливает; иначе явный red-flag и пересмотр стратегии зон до входа в M3. Tier 2 — отложенный пункт чек-листа (при доступе к GPU), не гейт CI.
- **Соответствие ТЗ:** Спринт 2 (нефункциональный критерий occupancy, A2). **Питает дизайн M3.**

- **Критерий (Tier-2, локально, не гейт облачного CI):** профиль ncu прототипного ядра без явного Thread Divergence; **achieved occupancy соответствует прогнозу M2.5/M2.7** — не ниже целевого порога, либо зафиксировано расхождение модель/факт с объяснением (A2); фактические регистры/поток внесены в отчёт зонда.

## M3.5 — TimeConveyor-CPU (логика TD без CUDA)  [ENG] [B6]
- **Цель (переквалифицирована 2026-06-11):** изолированная отладка конвейерной логики (B1–B3) + **постоянный конвейерный тест облачного CI** (ubuntu-latest без GPU) + CPU-референс для сверки с GPU-путём M4 (референс допусковый, не побитовый: libm и CUDA-math расходятся в последнем ulp). Исходный мотив B6 «критерий M4 нельзя прогнать локально» снят переездом на GPU-машину — веха сохраняется ради CI и отладки, **таймбокс**: CPU-производительность кольца не полировать.
- **Делается:** z программных узлов (`std::jthread` + SPSC-очереди как заглушка `ITransport`), ≥4 зон, оркестровка через `ZoneFSM`, чёт/нечёт I/O; **Δt-handoff** (B2/ZoneFSM §6: Δt следующего шага решает узел-отправитель по v_max последнего завершённого прохода и передаёт с первой зоной; правило тождественно при z=1); **PBC-замыкание** (B3/ZoneFSM §7.2: циклические индексы зон, партнёр-`d` со сдвигом шага, удержание первых зон); **фикс-точечное накопление сил** (B1). Допускается параллельность с подготовкой M4 (StreamTransport, events, фикс-точка на GPU); жёсткий гейт один — критерий M4 прогоняется только после зелёного M3.5.
- **Критерий приёмки:** `Test_determinism` — побитовое совпадение координат **и скоростей** (методика Гл. 3.6) 1 узел vs z узлов, в т.ч. при `timestep.mode=auto`; **репликация эксперимента §3.6** — Al-72, свободные границы, авто-шаг (C1=10 по диссертации), ≥25 900 шагов, побитово 1 vs z узлов (покрывает путь «кольцо без PBC-замыкания»); **ярлык: историческая реплика детерминизма конвейера, НЕ измерение плавления** (72 атома не имеют резкого перехода, см. `_meta/VALIDATION_EXPERIMENT_2026-06-12.md` §4); **сборка сил из частичных вкладов** — кросс-тест «сумма w-вкладов ≡ монолитный расчёт ≤1e-12» + тест однократности пары через границу зон (INV-8; защита от «читерской» реализации полным пересчётом); `Test_NVE_Invariant` 50 000 шагов: **секулярный тренд** $E_{full}$ (линейный фит, отделён от осцилляции Верле) в индустриальных единицах **kT/(ns·dof)** (реперы OpenMM, см. `_meta/MIXED_PRECISION_BESTPRACTICES_2026-06-11.md` §3.5: `deterministic_fp64` ориентир ~$10^{-7}$, `production_mixed` ~$2\cdot10^{-5}$ kT/(ns·dof)) при фиксированных dt=1 фс, T=300 K, режим точности указан — **многозонный конвейер не хуже 1-зонного эталона той же длины** (порог калибруется эталонным прогоном в начале вехи; абс. значение зависит от разрыва силы на $R_{cut}$ — см. аудит C1, решение force-shift/switching принять в M3); сохранение импульса; anti-deadlock ≥ 2z шагов; работает в CI. *(Старый порог «дрейф ≤1e-5 за 50k» отменён: измерено rel=1.46e-2 при 1 фс и dt-независимый пол из-за разрыва силы на кат-оффе.)*
- **Закрывает:** B1, B2, B3, B6; логическую часть A3. **Питает M4.**

## M4 — TimeConveyor: одно-GPU эмуляция (порт M3.5 на CUDA)
- **Цель:** перенос проверенной логики конвейера на массив `cudaStream_t`; первые честные перф-числа.
- **Делается:** `StreamTransport` (`cudaMemcpyAsync`, pinned-буферы — 2 copy-engines без page-locked памяти не утилизируются), аппаратная синхронизация `cudaEvent_t` (INV-2); **фикс-точечное накопление на GPU** (B1; FP64-`atomicAdd` — только как референс сверки: порядок не фиксирован и на consumer-Blackwell он ~38× медленнее int64); режим `deterministic_fp64` ⇒ FP64-offsets и транспорт (B5); **инстанциация `production_mixed`** (перенесено из M6: на 1:66.7 перф-замеры в FP64 измеряют пропускную способность двух FP64-юнитов/SM, а не дизайн конвейера); **дизайн-вопрос точности координат на handoff** (B5/MINT32: дрейф mixed гонят координаты, не FP32-силы — выбрать int32-фикс-точку offsets/double-single vs FP32 до фиксации формата зоны, см. `MIXED_PRECISION_BESTPRACTICES_2026-06-11.md` §3.4); verify-сборка с `--fmad=false` (детерминизм-сравнения CPU↔GPU); **бенчмарк-baseline «TD-конвейер vs наивный single-kernel путь»** + методика `docs/TD_MD_Core_Bench_v1_0.md` (atom-steps/s, N=10⁵–10⁶, режим `production_mixed`, прогрев/повторы/версии драйвера) — фиксация коэффициента и условий (A2), **не** pass/fail-гейт (ТЗ §1: цель — multi-device масштабирование малых систем).
- **Критерий (главный):** те же тесты, что в M3.5 (побитовый детерминизм 1 стрим vs N стримов — в `deterministic_fp64`; run-to-run побитово при фиксированном z — в `production_mixed`; NVE-инвариант по методике M3.5; репликация §3.6; PBC; anti-deadlock) — на GPU; `compute-sanitizer` (memcheck+racecheck) чистый на всех `cuda`-тестах (локальный гейт `scripts/gpu_gate.sh`); achieved occupancy зафиксирована и сопоставлена с прогнозом M2.5/M2.7; **все перф-числа аннотированы режимом точности**; контрольная точка памяти: N=10⁷ влезает в ≤3.5 GiB.
- **Закрывает:** A3 (single-GPU = валидация логики, не производительности).
- **Соответствие ТЗ:** Спринт 3.

## M5a — Multi-process кольцо на одном GPU (MPI, локально)
- **Цель:** реальный `MpiTransport` и кольцо из ОС-процессов без потери корректности — максимум достоверности, достижимый на одной физической GPU (A4).
- **Делается:** `MpiTransport` с **host-staging** (D2H → MPI → H2D) как базовый путь — системный OpenMPI 4.1.6 не CUDA-aware (`opal_built_with_cuda_support=false`, измерено), device-указатели в MPI не ходят; контракт `ITransport` фиксирует memory space; 2–4 ранга × 1 GPU (MPS/Default compute mode); **k временных шагов на узел** (`decomposition.ring.steps_per_node`, Гл. 3.4, ур. 49–51) — главный рычаг occupancy (B4: в полёте у узла до k зон; ориентир k ≈ zones_to_saturate из зонда), трафик ↓ в k раз.
- **Критерий:** детерминизм/NVE-тесты M4 проходят на ≥2 MPI-рангах; насыщение GPU при k>1 зафиксировано профилем nsys; издержки host-staging измерены и внесены в Bench-отчёт.
- **Соответствие ТЗ:** Спринт 3 (ступень A4).

## M5b — Multi-GPU кластер (реальное кольцо)  [внешнее предусловие: доступ к узлу с ≥2 GPU]
- **Цель:** физическое кольцо без потери корректности; масштабирование.
- **Делается:** `P2PTransport` (`cudaMemcpyPeerAsync` — где P2P есть; NVLink/GPUDirect из базовых допущений исключены: на consumer-Blackwell их нет) и CUDA-aware MPI (UCX) за runtime-детектом `MPIX_Query_cuda_support`; перенос k-шагового кольца на физические узлы.
- **Критерий:** корректность (детерминизм/NVE) сохраняется на ≥2 GPU; метрика **P2P Masking %** ненулевая; **эффективность масштабирования $E = t_1/(P\,t_z)$ (операционализация ур. 47)** с числовым порогом, зафиксированным до прогона (ориентир диссертации — линейный рост, >95%, Гл. 2.6; недостижение = red flag метода на GPU-транспорте, не «график снят»); strong/weak scaling на LJ-конфигурациях масштаба §3.5 (10⁶–4·10⁶ атомов) — **флагманский приёмочный прогон** (визитка), 10⁶–10⁷ атомов = single-GPU фронтир 2025 (LAMMPS-KOKKOS 16 млн LJ/H100); полная спецификация — `_meta/VALIDATION_EXPERIMENT_2026-06-12.md` §6.
- **Соответствие ТЗ:** Спринт 3 + 5.

## M6 — Physics Suite (многочастичные потенциалы)
- **Цель:** за пределы Морзе.
- **Предусловие (поставить заранее, можно с M3):** локальный LAMMPS (сборка с GPU-пакетом не обязательна) — без него критерий RDF буквально неисполним; одноразовая сверка golden-датасета с `reference_data/in.al_morse` (лежит с M0 неиспользованным); генерация эталонных RDF/траекторий для больших N.
- **Делается:** EAM (отдельное ядро), затем FS; **дизайн партиального накопления электронной плотности ρ** поверх w-механизма зон (двухпроходность ρ→F на границах зон: w-вклад соседа — это и плотность, и силы; B-ревью, M6); **табличные потенциалы со сглаживанием (Гл. 1.3)** — заявлены в ТЗ §5 и трассируемости, вехи-владельца до сих пор не имели; интерфейс ML-потенциалов (заглушка). *(production mixed-precision перенесён в M4.)*
- **Критерий:** EAM-силы и RDF длинной траектории vs LAMMPS — совпадение RDF $\ge99.5\%$ (`Test_Statistical`) **по зафиксированной заранее спецификации метрики** (биннинг, $r_{max}$, норма сравнения — записать в Bench/ConfigSchema до прогона: метрика без спецификации нефальсифицируема); протокол приёмки — **энергия + дрейф импульса на наборе систем** (образец HOOMD/Anderson 2008, см. `MIXED_PRECISION_BESTPRACTICES_2026-06-11.md` §2); **точку плавления мерить ТОЛЬКО двухфазным coexistence** (`interface/NVE`, работает в нашем NVE-only режиме), НЕ одно-фазным нагревом (перегрев +19%) и НЕ на парном Морзе/LJ (металлическое плавление врут) — обоснование и протокол в `_meta/VALIDATION_EXPERIMENT_2026-06-12.md` §5,§7. EAM в `production_mixed` чувствителен к ошибке плотности ρ (embedding-функция) — закладывать FP64-плотность или `deterministic_fp64` (рекомендация LAMMPS для many-body). Замечание: EAM/FS — `[ENG]`, в диссертации не валидировались (Гл. 3.2 — лишь обоснование возможности).
- **Соответствие ТЗ:** Спринт 4.

## M7 — UX / I/O / DevOps / HPC-деплой
- **Цель:** продуктовая зрелость.
- **Делается:** Live CLI Dashboard (ANSI, P2P Masking %, инварианты), асинхронный I/O в фоновом потоке, Rescue Dump на CUDA-ошибки, `Apptainer.def` + `spack.yaml`, полный CI (линт, сборка GCC/Clang, юнит-тесты VerifyLab).
- **Критерий:** CI зелёный на чистой машине (без LAMMPS — golden-тесты); деплой контейнера на кластере проходит smoke-тест.
- **Соответствие ТЗ:** Спринт 1 (CI/контейнеры) + 5.

## Сквозные требования (на каждой вехе)
- Каждый новый модуль — с юнит-тестом (см. `Skeleton_Interfaces §9`).
- `[ENG]`-решения не противоречат «источнику истины»; спорное — сверять с диссертацией.
- `seed` фиксирован; прогоны воспроизводимы.
- VerifyLab опционален за `-DWITH_LAMMPS=ON`; базовый CI — на golden-данных (A11).
- Локальная разработка — **Linux + RTX 5080** (sm_120, CUDA 13.1; ncu/nsys/compute-sanitizer локально; измеренный профиль — `_meta/ARCH_AUDIT_GPU_2026-06-11.md` §0). GPU-тесты несут метку `cuda` и гейтятся локально (`scripts/gpu_gate.sh`: ctest + compute-sanitizer); облачный CI — CPU-тесты + compile-only CUDA-job; self-hosted runner, если появится, — только `workflow_dispatch`/push в main (репо публичный: runner на `pull_request` = исполнение чужого кода на dev-машине). Кластер (≥2 GPU) — **внешнее предусловие только M5b**.
- Все перф-числа аннотируются режимом точности (`production_mixed`/`deterministic_fp64`): измеренный разрыв FP64:FP32 на целевом чипе — 1:66.7.
